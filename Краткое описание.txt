Краткое описание решения

Решение создано для учета и частичного анализа операци движения денежных средств подотчетных лиц.

Архитектура имеет в себе ключевые сущности:
- проекты;
- бюджетные статьи;
- данные о подотчетных лицах;
- таблицу операций.

В решении реализован доступ к данным через локальную груповую политику.
При создании операции она будет доступна всем пользователем из групп в которых состоит пользователь создавший операцию, а так же, все пользователям из групп помеченных CanReadAllData

Для редактирования данных о пользователях системы необходимо иметь выделенные права, помечены CanChangeUsersData, то есть, не каждый пользователь помечен IsAdmin может редактирвоать данные о пользователе.

Править группы пользователей, списки проектов, бюджетных статей и данных о подотчетных лицах может только пользователь помечен IsAdmin = true.
Выше указанные данные не подлежат удалению по причине сохранности исторических данных.

Операци, созданные пользователями, поддерживают все CRUD операции с учетом обрабтки конфликтов оптимистичного парралелизма.

На данный момент не реализован инсталятор, поэтому внести в базу данных первоначальную группу пользователей и учетную запись админа необходимо вручную с учетом  алгоритма шифрования паролей указанному в сущности User.
Кроме этого, в конфигурационных файлах хостинга службы (на данный момент используеться встроенный хостинг VS для служб WCF) и клиента надо указать корректный адрес в теге connectionString .... DataSource для службы и адрес конечной точки службы для клиента.

Метаданные службы включены.

Так же, можно отключить в конфигурационном файле службы <serviceDebug includeExceptionDetailInFaults="True" /> для отмены передачи информации об непредвиденных исключениях клиенту.
